# Phase 5.4 — Homebrew tap implementation plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` if more work needs to be done. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Auto-publish a Homebrew formula for `diffah` to a personal
tap (`leosocy/homebrew-tap`) on every release, so macOS / Linuxbrew
users can `brew install leosocy/tap/diffah` instead of pulling a
binary from the releases page.

**Architecture:** Adds a `brews:` block to the existing
`.goreleaser.yaml`. On every `goreleaser release --clean` run, the
goreleaser action templates a `Formula/diffah.rb` referencing the
just-built tar.gz archive (with sha256), then commits and pushes it
to `leosocy/homebrew-tap` using a personal-access token
(`HOMEBREW_TAP_TOKEN`) supplied as a workflow secret.

**Scope decisions:**

- **D1 — Tap repo name.** GitHub repo name is `leosocy/homebrew-tap`
  (Homebrew convention strips the `homebrew-` prefix when users
  reference it: `brew tap leosocy/tap`). The roadmap says "tap
  `leosocy/tap`" — same thing.
- **D2 — Direct push, not PR.** `goreleaser` commits directly to the
  tap repo's default branch. The tap is single-purpose; no review
  is needed for a single-file change generated by tooling.
- **D3 — `skip_upload: auto`.** Skip publishing on `-pre` / `-rc`
  tags so the tap reflects only stable releases.
- **D4 — Single binary, two architectures, two OSes.** The existing
  `archives:` block produces darwin-amd64, darwin-arm64,
  linux-amd64, linux-arm64 archives. The brews block must reference
  all four variants — goreleaser handles that automatically by
  inferring from `archives` + `goarm` patterns. No per-variant config
  needed.
- **D5 — Test stanza.** The formula's `test do` block runs
  `#{bin}/diffah version` and asserts non-empty output. That's the
  simplest sanity check that the binary is present and executable.
- **D6 — Caveats.** Mention that `zstd` is required only for some
  intra-layer modes (matches README's existing requirements section).
  Don't require it as a hard `depends_on` — many users won't need it.
- **D7 — License.** Apache-2.0 to match `LICENSE`.
- **D8 — README placement.** New "Homebrew" subsection inside the
  existing `## Install` block, between "Prebuilt binaries" and
  "Container image". Mention both the tap-installed path and the
  `--HEAD` install for tracking master.
- **D9 — Verification on next release.** A standalone post-release
  check, not a smoke test. Goreleaser runs only on tag push; we
  cannot trigger it from a PR. The PR description includes the
  verification checklist for the operator to run after the next
  tagged release.

**Out of scope:**

- Bottle (pre-built binary) generation — Homebrew downloads the
  release tar.gz at install time. Bottles can land later as a
  P5.4-followup if install time becomes a real complaint.
- Conflicts/depends_on for `zstd` — soft requirement only, as a
  caveat. Users who hit the requirement will see a clear error
  from `diffah` itself.
- A separate Linuxbrew formula. The existing `brews:` block
  already covers Linuxbrew via the same `goos: linux` archives.

## Manual prerequisites (operator action — cannot be automated by Claude)

Goreleaser cannot push to a non-existent repo, and a personal-access
token cannot be created from CI. These three steps must be done by
the repo owner before the next release tag will succeed:

- [ ] **MP1 — Create the tap repository.** On GitHub, create
      `leosocy/homebrew-tap` as a public repo with a README. Default
      branch `main`. No special settings beyond letting GitHub
      Actions tokens push (default).
- [ ] **MP2 — Generate the access token.** GitHub Settings →
      Developer settings → Personal access tokens → Fine-grained.
      Repository access: **only** `leosocy/homebrew-tap`.
      Permissions: Contents: **Read and write**. Metadata: Read.
      Expires: pick whatever your security policy mandates.
- [ ] **MP3 — Wire the token as a secret.** On
      `leosocy/diffah` repo → Settings → Secrets → Actions → New
      repository secret. Name: `HOMEBREW_TAP_TOKEN`. Value: the
      token from MP2.
- [ ] **MP4 — Cut the next release tag.** `git tag vX.Y.Z &&
      git push --tags`. Watch the release workflow; goreleaser
      pushes `Formula/diffah.rb` to the tap during the
      `release --clean` step.

If MP1–MP3 are not done before the next tag, the release workflow
will fail loudly at the goreleaser step (token missing or repo
404). The rest of the release (binaries, container image, signing)
proceeds as before because goreleaser's `brews:` is independent.
